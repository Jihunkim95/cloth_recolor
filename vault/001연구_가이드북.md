# Part 1: Obsidian 사용법

## Obsidian이 뭐야?

메모장이다. 단, 메모끼리 연결할 수 있는 메모장.

## 쓰는 방법

**메모 만들기:** 왼쪽 위 + 버튼 누르기. 끝.

**메모 연결하기:** 글 쓰다가 `[[다른 메모 이름]]` 이렇게 쓰면 자동으로 링크가 생긴다. 클릭하면 그 메모로 이동한다.

**읽기 모드:** `Ctrl + E`를 누르면 예쁘게 보인다. 다시 누르면 편집 모드.

**검색:** `Ctrl + Shift + F`를 누르면 모든 메모에서 단어를 찾는다.

## 우리가 쓰는 구조

```
vault/
├── Daily_루틴.md          ← 매일 연다
├── 연구_대시보드.md        ← 전체 현황 보기
├── 연구_가이드북.md        ← 지금 이 문서
├── notes/                 ← 개념 정리 노트들
│   ├── 렌더링_방정식.md
│   ├── BRDF.md
│   └── ...
└── results/               ← 실험 결과만 모아둔 곳
    ├── exp001.md
    ├── exp002.md
    └── ...
```

vault 밖에는:

```
~/research/                 ← VS Code로 열기. git은 여기서.
~/papers/                     ← PDF 보관용
```

**규칙: vault 안에는 .md 파일만 넣는다.** 코드, PDF, 이미지는 밖에 둔다.

## 노트 쓰는 법

모든 노트는 맨 위에 이것만 붙인다:

```yaml
---
title: 노트 제목
layer: 0 또는 1 또는 2 또는 1+2
date: 2026-02-13
---
```

- `layer: 0` = 절대 안 변하는 물리 법칙
- `layer: 1` = 물리 법칙에서 나온 결과
- `layer: 2` = 누군가 정한 방식 (깰 수 있다)
- `layer: 1+2` = 둘 다 섞여 있음

노트를 쓸 때마다 layer만 적으면, 대시보드가 알아서 분류해준다.

## 대시보드 보는 법

`연구_대시보드.md`를 읽기 모드로 열면 테이블이 자동으로 뜬다.
아무것도 안 해도 된다. 노트를 추가하면 알아서 반영된다.

## Q&A 규칙 (헷갈렸던 것 기록)

> **공부하다 헷갈렸던 Q&A 는 [[3dgs_reconstruction_overview]] 의 §5 FAQ 한 곳에만 쌓는다.**

- 각 노트에 FAQ 섹션 만들지 않는다 (관리 지옥).
- Q 가 너무 좁아서 overview 에 안 어울리면 → 그 노트 본문을 직접 수정해서 명료화 (Q&A 포맷 안 만듦).
- 결정 1개, 위치 1개. 끝.

# Part 2: 실험 하는 법

## 실험이란?

"이렇게 하면 결과가 좋아질까?"를 확인하는 것이다. 생각만 하면 모른다. 돌려봐야 안다.

## 실험 하나의 흐름

```
질문을 정한다
   ↓
코드를 짠다
   ↓
돌린다
   ↓
숫자를 본다
   ↓
결과를 적는다
   ↓
다음 질문을 정한다
```

이것이 계속 반복된다.

## 실험 시작 전에 반드시 적을 것

코드를 짜기 **전에** 이 세 가지를 먼저 쓴다:

```
1. 질문: 뭘 알고 싶은가?
2. 예측: 어떤 결과가 나올 것 같은가?
3. 기준: 어떤 숫자가 나오면 성공인가?
```

이걸 안 쓰면 실험이 끝나도 "그래서 이게 좋은 건가 나쁜 건가?"를 모르게 된다.

## 실험 결과 기록 방법

실험이 끝나면 터미널에서:

```bash
결과 exp001
```

이러면 `~/vault/results/exp001.md`가 만들어지고 VS Code에서 열린다.
(alias 설정은 아래 "터미널 설정" 참고)

열린 파일을 이렇게 채운다:

```markdown
---
title: exp001 — vanilla 3DGS RC 위반량 측정
status: done
date: 2026-02-15
code: ~/researh/experiments/exp001_RC_violation/
---

# exp001

## 질문
vanilla 3DGS가 reflection consistency를 얼마나 위반하는가?

## 설정
- 장면: Shiny Blender ball
- 모델: vanilla 3DGS, 30k iterations
- 측정: 정반사 표면 100개 점

## 예측
반사 방향이 비슷한 시점 쌍에서도 색상 차이가 클 것이다.

## 결과
- 평균 위반량: 0.15
- 최대 위반량: 0.42

## 예측 맞았나?
맞았다. 위반량이 유의미하게 크다.

## 다음
exp002: L_RC를 auxiliary loss로 추가해서 위반량이 줄어드는지 확인
```

**핵심: "예측"과 "예측 맞았나?"가 있어야 한다.** 이게 없으면 실험을 돌려놓고 의미를 모른다.

## 실험 이름 규칙

```
exp001, exp002, exp003 ...
```

번호만 붙인다. 날짜, 카테고리 같은 거 안 넣는다. 단순해야 안 깨진다.

## 터미널 설정 (한 번만 하면 됨)

`~/.bashrc` 또는 `~/.zshrc`에 이 한 줄을 추가:

```bash
결과(){ echo "---\ntitle: $1\nstatus: done\ndate: $(date +%Y-%m-%d)\ncode: \n---\n# $1\n## 질문\n\n## 설정\n\n## 예측\n\n## 결과\n\n## 예측 맞았나?\n\n## 다음\n" > ~/vault/results/$1.md && code ~/vault/results/$1.md; }
```

이후 터미널에서 `결과 exp001`만 치면 된다.

# Part 3: papers 구조

## papers이란?

"나는 이런 문제를 발견했고, 이렇게 풀었고, 이만큼 좋아졌다"를 증명하는 문서.

## CVPR papers의 뼈대

papers은 항상 이 순서다. 예외 없다.

### 1. Abstract (초록)

papers 전체를 5~6문장으로 요약. 이 구조를 따른다:

```
문장 1: 이 분야는 ~가 중요하다.
문장 2: 기존 방법들은 ~가 문제다.
문장 3: 우리는 ~를 제안한다.
문장 4: 구체적으로 ~를 한다.
문장 5: 실험 결과 ~만큼 좋아졌다.
문장 6: (선택) 코드를 공개한다.
```

### 2. Introduction (서론)

Abstract를 확장한 것. 1~2 페이지. 이 흐름을 따른다:

```
단락 1: 왜 이 문제가 중요한가 (배경)
단락 2: 기존 방법들은 뭘 했는가 (선행 연구 요약)
단락 3: 기존 방법의 한계는 무엇인가 (문제 정의)
단락 4: 우리의 핵심 아이디어는 무엇인가 (직관 설명)
단락 5: 우리의 기여(contributions)를 bullet으로 나열
```

contributions은 보통 3개:
- 새로운 방법/모델 제안
- 새로운 학습 전략/loss 제안
- SOTA 실험 결과

### 3. Related Work (관련 연구)

기존 papers들을 카테고리별로 정리. 우리 papers이 아닌 다른 사람들의 이야기.

3DGS 정반사 연구라면 보통 이렇게 나눈다:
- View-dependent appearance in 3DGS
- Physically-based rendering in neural/Gaussian fields
- Reflection modeling (environment map, ray tracing 등)

각 카테고리 마지막에 "그런데 ~는 아직 안 했다. 우리가 한다."로 마무리.

### 4. Method (방법)

papers의 핵심. "우리가 정확히 뭘 했는가"를 수식과 그림으로 설명.

```
4.1 전체 파이프라인 개요 (그림 1개 반드시 필요)
4.2 핵심 기법 1 (수식 + 직관 설명)
4.3 핵심 기법 2
4.4 학습 전략 / Loss function
```

**규칙:**
- 모든 수식에는 "왜 이렇게 하는가"가 한 문장으로 있어야 한다
- 파이프라인 그림은 왼쪽이 입력, 오른쪽이 출력
- 기호를 처음 쓸 때 반드시 정의한다

### 5. Experiments (실험)

"정말로 좋아졌는가"를 숫자로 증명.

```
5.1 실험 설정
    - 데이터셋: 뭘 썼는가
    - 비교 대상(baselines): 누구랑 비교하는가
    - 메트릭: PSNR, SSIM, LPIPS
    - 구현 디테일: GPU, 학습 시간, 해상도

5.2 정량적 비교 (테이블)
    - 숫자가 다른 방법보다 높다/낮다를 보여줌
    - bold로 best, underline으로 second best

5.3 정성적 비교 (그림)
    - 같은 장면을 여러 방법으로 렌더링한 결과를 나란히 배치
    - zoom-in box로 차이가 명확한 부분을 강조

5.4 Ablation Study
    - 우리 방법에서 각 component를 하나씩 빼면서 효과를 증명
    - "이걸 빼면 이만큼 나빠진다" = "이게 중요하다"
```

### 6. Conclusion (결론)

3~4문장으로 끝낸다:

```
문장 1: 우리는 ~를 제안했다 (한 줄 요약).
문장 2: 실험 결과 ~를 달성했다.
문장 3: 한계점은 ~이다 (솔직하게).
문장 4: 향후 ~를 할 수 있다.
```

## 페이지 배분 (8페이지 기준)

```
Abstract:     0.3 페이지
Introduction: 1.0 페이지
Related Work: 1.0 페이지
Method:       2.5 페이지  ← 가장 많아야 한다
Experiments:  2.5 페이지  ← 이것도 많아야 한다
Conclusion:   0.3 페이지
References:   별도 (페이지 제한에 안 들어감)
```

Method와 Experiments가 합쳐서 5페이지. papers의 62%. 여기가 약하면 reject.

## 그림 배치

```
Figure 1: 파이프라인 전체 그림 (Method 앞쪽)
Figure 2: 핵심 기법 상세 그림 (Method 중간)
Figure 3: 정성적 비교 (Experiments)
Figure 4: Ablation 시각화 (Experiments)
```

최소 4개. 그림이 부족하면 papers이 텍스트 벽이 되어서 리뷰어가 읽기 싫어한다.

## papers 작성 순서

글을 쓰는 순서는 papers의 순서가 아니다:

```
1번째: Experiments를 먼저 쓴다 (결과가 있어야 papers이 있다)
2번째: Method를 쓴다 (결과를 만든 방법을 설명)
3번째: Introduction을 쓴다 (왜 이 방법이 필요한지)
4번째: Related Work를 쓴다 (다른 사람들이 뭘 했는지)
5번째: Abstract를 쓴다 (전체를 요약)
6번째: Conclusion을 쓴다 (마무리)
```

**Abstract를 먼저 쓰지 않는다.** 실험 결과가 없으면 뭘 요약할 것인가.

# 전체 흐름 한눈에 보기

```
매일 아침
  → Daily 루틴 열기
  → 오늘 할 것 한 가지 정하기
  → 90분 집중

실험할 때
  → 질문, 예측, 기준을 먼저 쓴다
  → 코드 짜고 돌린다
  → 끝나면 "결과 expXXX" 치고 결과를 적는다

노트 쓸 때
  → layer 적는다
  → "깰 수 있는 것 → 깨면?" 을 쓴다
  → 대시보드에 알아서 반영됨

papers 쓸 때
  → Experiments 먼저
  → Method → Intro → Related → Abstract → Conclusion 순서
```
